# Why not X? — Portable AI Assets System Differentiators

## Short answer

Most adjacent tools solve **one layer** of the problem.
Portable AI Assets System is focused on the **ownership + portability layer** above them.

---

## Why not Dify / Flowise / Langflow?

These tools are strong at:
- workflow building
- orchestration
- visual pipelines
- app-style AI composition

They are usually **not optimized for**:
- canonical personal/team AI asset ownership
- cross-runtime adapter projection
- drift-aware migration between already-used machines
- review/apply pipelines for runtime-local instruction files
- Git-friendly long-lived AI asset continuity as the main product thesis

Portable AI Assets System is not trying to replace workflow builders.
It aims to make the user's AI assets portable **across** such systems.

---

## Why not MemGPT / Letta / memory frameworks?

Memory systems are good at:
- persistent memory
- context recall
- layered memory architectures
- retrieval-backed continuity

They are usually not the full answer for:
- skills
- prompts
- workflows
- adapters
- tool bindings
- machine bootstrap and restore
- reviewable migration across different runtimes

Portable AI Assets System treats memory as one important asset class — not the whole stack.

---

## Why not MCP?

MCP is important because it helps standardize how models access:
- tools
- resources
- external context

But MCP is mainly a **tool/resource interoperability protocol**.
It does not, by itself, define:
- portable user profile schema
- long-lived AI asset packs
- review/apply migration workflows
- cross-machine continuity for prompts, skills, adapters, and summaries

Portable AI Assets System can use MCP, but is not replaced by MCP.

---

## Why not just use a single platform's projects / assistants / GPTs?

Single-platform assistants are convenient, but they usually imply:
- platform lock-in
- weak exportability
- weak auditability
- weak portability across clients and machines

Portable AI Assets System is built around the opposite assumption:

> the user should own the AI asset layer, not any one runtime.

---

## The core differentiator

Portable AI Assets System focuses on this combination:

- canonical source of truth
- adapter projections into runtime-specific surfaces
- Git layer + non-Git backup layer
- drift-aware inspect / diff / review / apply pipeline
- cross-agent, cross-machine continuity

That combination is the differentiator.
